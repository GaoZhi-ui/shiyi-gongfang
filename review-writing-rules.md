## 1. 基类设计

### WritingRule（抽象基类）

**好的设计：**

- `__init_subclass__` 自动校验非抽象子类的 `name` 和 `description`，避免遗漏必要属性。这个钩子时机正确——在类创建时执行，不会被延后到实例化时才发现。
- `to_dict()` 提供统一的元数据序列化接口，路由层直接调用，松耦合。
- `severity_level` 属性提供排序权重，思路清晰。

**隐患：**

1. **`severity` 类型约束缺失**。`severity` 只是 `str`，未使用 `Literal["error", "warning", "info"]`。子类可以写成 `severity = "critical"`，`severity_level` 会静默回退到 1（info 级别），导致严重问题被降级。如果路由层或 UI 依赖这三级做展示，传入无效值不会报错，只在排序时悄无声息地变形。

2. **`WritingIssue` 字段未做校验**。`line` 字段类型为 `int`，但如果子类传入 0 或负数，没有约束。`line` 的语义是 1-indexed，但 0 不会被拒绝。

3. **`WritingIssue` 的 `severity` 也是裸 `str`**，同上，没有枚举约束。

### RuleRegistry（单例）

**隐患：**

4. **线程不安全**。`RuleRegistry` 是裸单例，`_rules` 字典没有任何锁保护。FastAPI 在单进程多线程模式下，`check_all` 遍历 `self._rules.values()` 的同时如果另一线程调用了 `register`/`unregister`，可能导致 `RuntimeError: dictionary changed size during iteration`。生产级 FastAPI 应用通常以多 worker 方式运行，但即便单 worker 多线程，字典并发修改也是竞态条件。

   修复思路：注册/注销只发生在启动期，运行时不做变更——但当前设计没有禁止运行时调用 `register`/`unregister`，也没有文档约束。

5. **单例与测试冲突**。`_instance` 是类级缓存，`__new__` 完全阻止了多实例构造。如果单元测试需要隔离的 registry（比如一个测试注册了一些假规则，另一个测试希望干净的注册表），两个测试会共享同一个 `_instance`，互相污染。常见的测试方案包括：提供一个 `_reset()` 方法（但当前没有），或者改用依赖注入而非单例。

6. **`check_all` 的排序 sort key 冗余**。排序使用 `{"error": 3, "warning": 2, "info": 1}.get(...)` 作为负值降序排列，但这个字典在 `severity_level` 属性和排序中重复定义了两次。如果未来扩展 severity 级别，需要修改两处，容易遗漏。应改为复用 `rule.severity_level`。

   当前写法：`key=lambda i: (i.line, -{"error": 3, "warning": 2, "info": 1}.get(i.severity, 1))`
   改进：`key=lambda i: (i.line, -registry.get(i.rule_name).severity_level if registry.get(i.rule_name) else 0)`

   不过依赖 registry 查找又有新的耦合问题。更好的方案：在 `WritingIssue` 上直接存储 `severity_level`，或者在排序时不依赖规则名称回溯查找等级。

## 2. 自动发现机制

### `discover_and_register`

**好的设计：**

- 使用 `pkgutil.iter_modules` 而非简单的文件 glob，能正确处理包内嵌套。
- 在 `inspect.getmembers` 中加入了 `not getattr(obj, "__abstractmethods__", frozenset())` 过滤，避免实例化抽象中间类。

**问题：**

7. **`_path_to_package` 有多层嵌套包名丢失 bug**。

   ```python
   # 当前实现：
   return ".".join([parent.name] + list(rel.parts))
   ```

   假设包结构为 `a.b.c.rules`，`__init__.py` 在 `a/b/c/` 下。遍历 `parents` 时，第一个找到 `__init__.py` 的父目录是 `a/b/c`，`parent.name` 是 `c`，`rel` 是 `rules/`，组合结果是 `"c.rules"`，正确包名应为 `"a.b.c.rules"`。

   这个 bug 在当前三层结构（`core.rules`）中没有暴露，因为 `core` 恰好是顶层包。如果将来规则包移入更深层路径，或复用此函数，就会导入失败。

8. **静默跳过导入失败的模块**。

   ```python
   except Exception as e:
       continue  # 静默跳过无法导入的模块
   ```

   如果某个规则文件有语法错误、缺少依赖或导入副作用抛异常，开发者在启动日志中看不到任何线索。规则"消失"了，不会报错，也不会生效。应至少使用 `logging.warning` 记录。

9. **缺少 `__init__.py` 存在性检查**。`discover_and_register` 的 docstring 说"扫描指定目录/包"，但如果目录下没有 `__init__.py`，`importlib.import_module` 在 Python 3.3+ 的 namespace package 机制下可能不报错，但行为不确定。应在入口处显式检查 `__init__.py` 是否存在，如果不存在则抛出明确异常，而非让 `pkgutil.iter_modules` 静默返回空结果。

## 3. 各规则正则分析

### 3.1 filler_words.py

| 模式 | 风险 |
|------|------|
| `(突然\|然后\|其实\|竟然\|忽然\|似乎\|\n仿佛\|有点\|有些\|开始)` | **中等误报率**。"开始工作"中的"开始"是必要语义，不是填充词；"有点意思"是固定搭配。"其实"在议论文中作为转折逻辑词也有必要性。 |
| 无词边界约束 | "大概情况"会匹配"大概"、"开始计划"会匹配"开始"，这些在一般写作中是正常词汇。 |

**建议**：填充词检测本身就有"宁可误报也要提醒作者自己判断"的合理性，但应在 `suggestion` 中加上"如果该词承载了必要语意，可以忽略此提示"。

### 3.2 redundant_modifiers.py

| 模式 | 风险 |
|------|------|
| `(非常\|极其\|太\|很\|十分\|\n特别\|相当\|无比\|极度)\s*[\u4e00-\u9fff]+` | **高误报风险**。"太"匹配"太阳""太太""太空""太和殿""大古太"等；"特别"匹配"特别行政区""特别行动"等专有名词。 |
| 无词边界限制 | "相当"可匹配"相当初"（虽然"相当初"不常见），但"非常"不会匹配"非常规"因为"非常"后面跟的是"\s*"，但"非常规"中"非"是汉字... 不对，"非常"匹配"非常规"中的"非常"，而"规"是汉字，所以"非常规"整体会匹配。但"非常规"中的"非常"不是冗余修饰，是"常"的否定。 |
| 仅限中文形容词 | 英文形容词（very good）不会被检测——这对中文写作检查工具不算缺陷，但需要明确适用范围。 |

**建议**：
- "太""特别""相当" 这三个词歧义过高，应考虑从默认匹配列表中移除，或添加白名单排除模式。
- 副词后面跟的名词（如"非常时期"中的"时期"）不是冗余修饰，当前无法区分名词和形容词。

### 3.3 weak_words.py

| 模式 | 风险 |
|------|------|
| `(觉得\|感到\|认为\|好像\|也许\|大概\|可能)` | **中等误报率**。"可能性""可能存在""大概情况"中的这些词是合理表达，不一定需要删除。 |
| 无词边界 | "感到"在"感到荣幸"中可接受，"认为"在学术写作中是标准表述而非弱词。 |

**建议**：这个规则的核心问题是"弱词"的判断高度依赖语境。可以用同一个正则，但 `suggestion` 需要针对不同词给出不同建议，而不是统一模板。

### 3.4 passive_voice.py

| 模式 | 风险 |
|------|------|
| `被[\u4e00-\u9fff]+` | **漏报率较高**。常见被字句结构有两种：(1) 被 + 动词（"被打"）— 能匹配；(2) 被 + 名词/代词 + 动词（"被他打了"）— 也能匹配，因为"他打了"都是汉字。 |
| 遗漏"被"后面有空格的用法 | 如"被 发现"（少见但可能）无法匹配 `/被[\u4e00-\u9fff]+/`。 |
| 遗漏"被"与动词之间有助词 | "被狠狠地打了"——"狠狠地"中的"地"不是汉字（它是 CJK 符号，不在 `\u4e00-\u9fff` 范围内），"被狠狠地"无法匹配。 |
| 其他被动标记 | "受到""遭到""予以""加以"等也是中文常见的被动/使动表达，但当前规则只检测"被"字。 |

**建议**：
- 正则改为 `被[^。！？\n]{1,15}` 匹配"被"之后的一段文本（包括非汉字字符），然后用 `findall` 提取整段。或者分两步：先找包含"被"的句子，再提取匹配段。
- 考虑扩展检测范围，覆盖"受到""遭到""得以"等被动/使动形式。

### 3.5 long_sentence.py

这个规则的问题最多。

**正则拆分不完整：**

```python
_SENTENCE_SPLIT = re.compile(r"[^。！？\n]+[。！？\n]")
```

10. **最后一句如果没有句尾标点则不会被检测**。文本末尾的句子若没有句号/感叹号/问号/换行符，完全正则匹配不到。比如 `"今天天气很好我吃了饭然后去散步"`（没有句号）——这个长句不会触发任何告警。

11. **句尾包含换行符导致空句子被检测**。当连续两个换行符出现，`[^。！？\n]+\n` 会匹配空行前的换行符？不对，`[。！？\n]` 匹配换行符，但 `[^。！？\n]+` 要求至少一个非句尾字符，所以空行不会被匹配成一个句子。但如果一行只有句号，如 `"。\n"`，`[。！？\n]` 只匹配了一个字符，前面 `[^。！？\n]+` 不能匹配空字符串所以这个也不匹配。好，实际上空行不会被匹配到。但最后一行没有换行也没有句尾标点的问题确实存在。

12. **`_find_line` 只找 target 的第一次出现**。

    ```python
    @staticmethod
    def _find_line(text: str, target: str) -> int:
        idx = text.find(target)
        if idx == -1:
            return 1
        return text[:idx].count("\n") + 1
    ```

    如果文本中有两个相同的句子，`text.find(target)` 总返回第一个出现的索引。这意味着所有第二次出现的超长句子的行号都指向第一次出现的位置。这是明确的 bug。

13. **40 中文字阈值硬编码**。`_MAX_CJK = 40` 是模块级常量。不同文体对句子长度的容忍度不同（小说 vs 学术论文），应支持按规则配置或在请求参数中传入。

14. **全角/半角标点混用场景**。中文文本中偶尔混入英文句点 `.`、逗号 `,`、问号 `?`，正则 `[。！？\n]` 只匹配中英文标点，英文句点不会触发断句。如果一个中文长句中混用了英文逗号和句点，正则仍会尝试找到中文句号来断句——可能导致一个实际上的长句被当做一整段处理。

## 4. 路由层向后兼容性

**好的设计：**

- `_issue_to_old_format` 将 `rule_name` → `rule`，明确标记为兼容旧格式。如果旧客户端依赖 `rule` 字段名，这个转换保证了兼容。
- 输入校验做了：空文本拒绝、未知规则名拒绝、有效规则列表做差集校验。粒度合理。
- `body.rules is not None` 的处理区分了"传入空列表"和"不传"两种情况。

**潜在问题：**

15. **旧 API 可能期望不同的响应结构**。`_issue_to_old_format` 只做了 `rule_name` → `rule` 的字段名重映射。如果旧格式还有其他差异（比如字段名大小写、嵌套层级），这段逻辑覆盖不全。建议如果有可能，找一份旧 API 的响应样本比对。

16. **`StyleCheckBody` 对 `text` 的 `min_length=1` 校验**。Pydantic 的 `min_length` 在 `str` 上检查长度，空字符串 `""` 会被拒绝在路由层。但全空格字符串 `"   "` 长度 > 0，能通过 Pydantic 校验，直到路由函数中 `body.text.strip()` 才被拒绝。建议在 Pydantic model 上增加一个 validator 或使用 `Field(..., min_length=1, ...)` 加上业务层逻辑。当前是两段式校验，功能正确但逻辑分散。

17. **返回结构的 `rules_applied` 字段**。当 `body.rules` 是 `None` 时，`sorted(valid_rules)` 按名字排序返回所有规则。但注意字典在 Python 3.7+ 是插入有序的，`registry.list()` 保持规则被发现时的顺序，而 `sorted()` 改变了这个顺序。这不是 bug，但调用方可能期望按注册顺序而非字母顺序。需要确认前端是否依赖特定顺序。

## 5. 边界情况

| 场景 | 当前表现 | 评价 |
|------|----------|------|
| 空字符串 `""` | Pydantic min_length 拒绝，状态码 422 | OK |
| 纯空格 `"   "` | Pydantic 放行 → 路由层检验后 400 | OK |
| 纯英文文本 | 所有中文正则无匹配，返回 `[]` | OK |
| 非常长的文本（10000 行） | 每行循环，O(n) 扫描，无渐进式加载 | 性能 OK |
| 同字符串多处出现 | `_find_line` 只返回第一个行号 | **有 bug** |
| 文本不含句尾标点 | `long_sentence.py` 最后一个句子被忽略 | **有 bug** |
| 生僻汉字（CJK 扩展 B） | `_RE_CJK` 不包含扩展 B 范围 | 极少出现，可接受 |
| 全角空格（U+3000） | 不会被 strip 处理（Python str.strip 默认只 strip ASCII 空格） | **潜在问题**：中文文本中全角空格常见 |
| 零宽空格（U+200B） | 在 text 中但用户看不到，行号偏移不准确 | 极少出现 |
| HTML/XML 标签 | 文本中包含时会被当做普通文本处理 | 如果用户传入富文本，会产生误报 |

**补充发现：**

18. **全角空格问题**。中文文本中全角空格（U+3000 `　`）很常见。Python 的 `str.strip()` 默认只去除 ASCII 空格（` \t\n\r\x0b\x0c`），不会去除全角空格。因此 `if not stripped:` 不会识别全角空格行，这些行会进入正则匹配，虽然不太可能匹配中文关键词，但行号统计中包含了全角空格行。

19. **每行扫描的效率**。当前 5 个规则各自逐行扫描文本。如果有 10+ 规则，每个规则都做一次 `text.split("\n")` + `for line in lines`。对于大文本，可以共享行分割结果。不过当前规模下不是瓶颈。

## 总结

**正向评价：**

- `__init_subclass__` 校验的做法很成熟，比运行时检查更早暴露问题。
- 自动发现机制使用 `pkgutil` + `inspect`，比文件 glob + importlib 更可靠。
- 路由层的向后兼容转换意图正确，`_issue_to_old_format` 函数名清晰标明目的。
- 空文本有双层防护（路由层 + 注册中心层），设计上考虑周全。
- 每个规则文件单文件单规则，结构清晰，新增规则只需创建文件无需修改其他代码。

**核心薄弱点：**

最大的两个问题都在 `long_sentence.py`：**最后一句不检测**是功能性遗漏，**`_find_line` 只返回首次匹配行号**是明确的逻辑 bug。前者导致文本末尾的长句静默通过，后者导致重复句子的定位完全错误。

其次是自动发现机制的健壮性——静默吞异常和包名推导局限，在规则扩充时容易引入定位困难的 bug。

单例模式与测试的冲突在项目规模扩大后会成为阻塞因素，建议在早期就添加 `_reset()` 方法或提供工厂函数替代单例模式。
