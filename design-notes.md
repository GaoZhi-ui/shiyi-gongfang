# 写作助手工坊 —— 设计笔记

> 基于 prototype.html 的分析与延伸思考
> 日期：2026-05-20

---

## 一、总体印象

"写作助手工坊"定位为《泰拉拾遗录》的专属写作助手。整个产品可以理解为一个 **嵌入写作流程的 AI 伴侣界面**——不是通用的写作软件，而是围绕一个特定创作项目、一套固定写作方法论（9阶段流程 + 12项自检 + v3 风格准则）构建的工具。

现行原型是一个三栏布局的单页应用：左侧工具箱，中间聊天主区，右侧智能面板。设计语言主打暗色 + 金色高亮，冷峻但不冰冷。

---

## 二、布局分析

### 2.1 三栏结构

| 区域 | 宽度 | 角色 |
|------|------|------|
| 左侧 Sidebar | 280px | 功能导航 & 工具入口 |
| 中间 Main | 弹性 | 人机对话核心区 |
| 右侧 Panel | 360px | 上下文面板（当前自检项/审查结果/知识条目） |

这个比例合理。左侧固定宽度足够放下嵌套层级，右侧 360px 在 1920 宽屏上预留了充足的主区给对话。但在 1440 以下屏幕（MagicBook 的分辨率）上右侧会略挤，未来可以考虑可收起或切换显示。

### 2.2 主区（Chat）

对话式的交互范式对写作辅助来说是个好选择——写作者的工作流天然是交错的：写一句、停下来思考、提问、修改。Chat 界面比传统"打开一个文件开始写"更低阻。

但有一个结构性问题：**这个聊天区的角色是"和助手对话"，而不是"写作"本身**。也就是说，用户在聊天框里输入"检查当前章节"，助手返回审查结果——但用户真正在写的文字放哪里？原型没有提供一个"编辑器"区域。这意味着用户需要在外部（也许是 Typora、VS Code）写正文，然后粘贴进来让助手分析。

这不一定是个缺陷（如果工具定位就是"写作伴侣"而非"写作软件"），但它决定了产品形态的上限。如果未来想让用户真正在工坊里写完一章，需要一个编辑器标签页。

### 2.3 右侧 Panel

当前展示的是自检清单，这是一个"此时此刻"的上下文面板。它的角色是"不打断对话流的辅助信息展示"。这个设计很克制——没有把自检清单做成弹窗或独立页面，而是放在右侧让用户写作时随时瞄一眼。

## 三、视觉设计

### 3.1 色彩

- 底色：`#111`（主背景）、`#1a1a1a`（sidebar/panel 次级背景）、`#1e1e1e`（输入框）
- 边框：`#2a2a2a`——三层背景之间的隔断极轻，几乎融为一体，风格统一
- 强调色：`#e8a84c`——金色，温暖且有品质感。用在标题、hover 态、按钮上

配色方向是对的。暗色主题对写作者友好（减少亮屏疲劳），金色给了品牌记忆点。需要考虑的点：金色在 `#111` 背景上的可读性需要保证不刺眼，当前 `#e8a84c` 的亮度适中，OK。

### 3.2 字号/间距

- 标题 16px，工具项 13px，分组标题 11px，底部状态栏 11px
- 内边距统一使用 8/12/20 的倍数体系（8px 间距基础单位）

层次分明但间距偏大，考虑到侧栏只有 280px，12px 的工具项 + 8px 垂直间距 + 6px 圆角，信息密度控制得不错。

### 3.3 圆角

- 侧栏工具项：6px
- 输入框 / 发送按钮：8px
- 头像：6px

圆角体系简洁，6px / 8px 两个层级，没有过度使用。

### 3.4 对比度问题

`.tool-group-title` 使用 `color: #555` 在 `#1a1a1a` 背景上，按 WCAG AA 标准（4.5:1）可能略偏低。#555 的亮度约为 119，在 #1a1a1a（亮度约 52）上对比度大约在 3.5:1 左右。如果用户视力正常可能 OK（小字文本通常需要更强的对比度），建议微调到 #777 或提高亮度到 #888 以上。

## 四、功能拆解

### 4.1 侧栏工具分组

```
写作流程
  ├── 📋 9阶段流程
  ├── ✅ 自检清单
  └── 🔍 自动审查

知识库
  ├── 📚 工具箱一览
  ├── ✍️ 写作风格速查
  └── 👥 人物档案

章节管理
  ├── 🗺️ 全卷细纲
  ├── 📅 时间线表
  └── 🔗 伏笔追踪
```

9 个入口，3 个分组。覆盖了从写作流程（怎么做）→ 知识库（用什么）→ 章节管理（管什么）的全链路。结构清楚。

但有一个缺失：**当前没有"开始写作"或"打开章节草稿"的入口**。用户要怎么把一章的草稿交给助手？要么在对话里粘贴，要么需要在侧栏加一个"当前草稿"入口。

### 4.2 Topbar

显示"卷二 · 归途未定"——当前工作上下文。右侧有一个"⚙ API设置"。对单人项目来说这个 topbar 还行，但如果未来管理多卷，可能需要在 topbar 加卷/章节切换。

### 4.3 Chat 引导消息

> "你好，我是写作助手工坊。"
> • 检查当前章节
> • 运行审查
> • 调取知识库

三条引导清晰。但"检查当前章节"这个动作隐含了一个前提——用户已经把某个章节设为"当前"。当前界面上没有地方设定当前章节。要么是在对话中用指令设定（"我当前在写第31章"），要么需要设计一个"设置当前工作目标"的入口。

### 4.4 右侧面板：自检清单

```
场景首尾测试
展示不告诉测试
"他心想"压强测试
字数2000+
密度≤6
"不是A是B"≤2
"他"≤15/千字
```

这些直接来自 terra-writing-skill 的 12 项自检。清单本身是对的。但只要用户在右侧看到的是静态列表——如果它能在审查运行后动态显示每项的通过/失败状态，会更有用。比如：

```
✅ 场景首尾测试
❌ 展示不告诉测试 (第3段: "沈默意识到这是..." → 应展示)
✅ "他心想"压强测试
⚠️ 字数2000+ (当前: 1872 字)
...
```

把自检清单从一个"备忘"升级为"审查结果面板"，是这个右侧 Panel 最自然的演化方向。

## 五、交互问题与改进建议

### 5.1 核心交互流

当前用户的工作流大概是：

```
打开工坊 → 从外部编辑器粘贴章节正文 → 输入"检查" → 
助手返回审查结果 → 根据反馈回到外部编辑器修改 → 来回切换
```

如果在右侧 Panel 中嵌入一个**精简编辑器**（哪怕只显示当前章节纯文本、可原位修改），就能减少用户在不同窗口间的跳转。当然这增加了复杂度，需要掂量。

### 5.2 发送按钮

当前发送按钮是一个 `↵` 符号 + 金黄色背景。风格上配合金色主题，但需要注意：
- 按钮没有 hover 态（原型中没有定义），实际实现应该加上
- 文本区的 placeholder "输入写作指令…"——用户可能会在指令里粘贴大段正文，textarea 的 `rows=2` 有点少，建议允许自动增长

### 5.3 空状态

目前只有一个引导消息。以下场景缺少空状态设计：
- 侧栏工具点击后主区无响应（还没有绑定行为）
- "右侧面板没有内容时"——虽然自检清单默认显示，但点击"人物档案"等工具时右侧的变化逻辑没有定义
- 聊天区的历史消息超过一屏后的滚动锚定

### 5.4 移动端/窄屏

三栏布局在窄屏上无法工作。虽然当前定位是桌面端工作流（写作通常发生在桌面），但未来如果考虑平板或笔记本小屏，需要一套断点方案。最简单的做法：右侧 Panel 在 <1366px 时变为切换式浮层。

## 六、未来可能的扩展方向

1. **编辑器标签页**：在聊天区上方或侧栏增加"打开草稿"标签页，让用户直接在工坊里写作
2. **审查结果可视化**：右侧 Panel 变为动态审查 Dashboard，展示每一章的通过/失败项、字数变化曲线、伏笔状态
3. **多卷管理**：Topbar 增加卷/章节下拉切换，侧栏"章节管理"分组变为当前卷的章节树
4. **版本历史**：每次审查/修改的版本记录，可以回溯对比
5. **导出集成**：一键 MD → docx（对接现有的 terra-writing-skill 流水线）

---

## 七、总结

当前原型是一个完成度不错的 MVP 静态稿。它的核心优点在于：

- **清晰的定位**：不是通用写作工具，而是《泰拉拾遗录》的专属写作伴侣
- **克制的设计**：暗色 + 金色，三栏布局，功能分组清楚
- **对话式交互**：降低了与"写作助手"互动的心理门槛

最大的未决问题是 **"写作在哪里发生"**——用户需要自己在工具和编辑器之间切换。如果产品的边界就是"AI 写作伴侣"而非"写作软件"，那这个边界是合理的。但如果想让用户更沉浸，下一步就该考虑嵌入轻量编辑器。

原型中的每一个元素都有逻辑依据，没有无意义的装饰。这是它最可贵的地方。
