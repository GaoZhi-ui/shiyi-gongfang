# 拾遗工坊 · 安全审查报告

> 审查日期：2026-05-20
> 审查范围：拾遗工坊 FastAPI 应用（v1.0）
> 审查目标：本地写作辅助工具，绑定 127.0.0.1:8000

---

## 1. 数据安全

### 1.1 API Key 加密存储（Fernet）

**实现方式**（`services/key_manager.py`）：

- 用 PBKDF2HMAC（SHA-256, 100K iterations）从主机指纹派生 Fernet 密钥
- 指纹组合：`platform.node() + platform.machine() + USERNAME`
- 加密后 key 写入 `config.json`（被 `.gitignore` 排除，不进入版本控制）

**评估**：及格，但有三处硬伤

| # | 问题 | 严重程度 | 说明 |
|---|------|----------|------|
| 1 | **salt 为硬编码常量** | 中 | `b"shiyigongfang"` 写在代码里，等于降低了 KDF 的抗预计算能力。虽不影响运行时安全，但违背了 PBKDF2 的最佳实践——salt 应该随机生成、每个密钥独立 |
| 2 | **无密钥轮换机制** | 低 | API Key 一旦存入，除非手动删除重设，永远不会重新加密。如果密钥派生因子（主机名/用户名）意外变化，config.json 里的 key 直接不可恢复 |
| 3 | **本地攻击者可还原** | 中 | 三样指纹（主机名、架构、用户名）对拥有本地读写权限的进程来说唾手可得。Fernet 加密对于远程攻击是有效的屏障，对本地非管理员进程（如同机恶意脚本）也有一定防护；但对 admin 级入侵者，这层加密约等于明文 |

**结论**：对于本机写作工具这个威胁模型，方案合理。但如果项目未来需支持多机同步或云端部署，这套机器绑定的派生方案会带来迁移问题，建议改用用户设置的主密码派生。

### 1.2 config.json / .env 泄露风险

| 文件 | 状态 | 风险 |
|------|------|------|
| `config.json` | 在 `.gitignore` 中 | 安全。不会被 git 追踪。但如果 config.json 被备份工具（如同步盘、全盘备份）捕获，Fernet 加密提供了一层保护 |
| `.env` | 在 `.gitignore` 中 | 安全。不会进入版本控制。当前不存在于文件系统中 |
| `config.yaml` | 不存在 | N/A。api-design.md 中提及的设计尚未实现 |

**补充发现**：`routers/health.py` 会暴露 `"has_api_key": true/false` 信息。虽然不泄露密钥原文，但告诉攻击者"这个服务配置了 Key"——信息泄露，低风险。

## 2. 文件安全

### 2.1 路径穿越防护

四个文件操作路径都实现了防护：

| 路由模块 | 防护函数 | 检查项 |
|----------|----------|--------|
| `routers/knowledge.py` | `safe_resolve()` | `.resolve()` 规范化 → 前缀校验 → 扩展白名单 → 存在检查 → 5MB 上限 |
| `routers/chapters.py` | `_safe_chapter_path()` | 同上，但只允许 `.md` |
| `routers/chat.py` | `_read_knowledge_file()` | 独立复制了路径校验（`resolve` + 前缀 + `.md/.txt` + 1MB） |
| `routers/tools.py` | subprocess `cwd` + 路径校验 | 脚本白名单 + 参数正则 + 文件路径二次校验 |

**评估**：到位。核心检查逻辑是正确的：

- `.resolve()` 消除了 `..` 和符号链接
- `startswith(str(base_dir.resolve()))` 确保不能越界
- 后缀白名单限制了可读文件类型

**遗留风险**：

| # | 问题 | 风险 | 说明 |
|---|------|------|------|
| 1 | TOCTOU 竞态 | 极低 | 检查 → 读取之间文件可被替换为符号链接。本地单用户场景基本不可利用 |
| 2 | `safe_write()` 缺少文件大小上限 | 低 | 写入 route 没有 5MB 限制。FastAPI 默认 body 上限 ~16MB，但无显式约束。恶意或意外大写入可能撑满磁盘 |
| 3 | 知识库路径解析回退链不够严格 | 低 | `_resolve_knowledge_base()` 回退到 `E:/openhanako-work/knowledge_base/泰拉拾遗录` 是硬编码路径，会跳过校验 |
| 4 | 文件列表 API 会读取所有文件内容做 CJK 计数 | 低 | `list_files()` 在 GET /knowledge 时读完每个文件做正则匹配，如果是海量文件（>1000），会有性能问题 |

### 2.2 知识库文件越权读取

API 设计了统一的入口（GET /knowledge/{filepath}），没有用户认证层。但：

- 服务绑定 `127.0.0.1`，仅本机可访问
- CORS 设置 `allow_origins=["*"]`（与 auth 无关，仅影响浏览器跨域限制）
- 路径穿越被阻断

**结论**：对本机工具来说，不存在"越权"问题——访问者即是机器所有者。如果未来加入多用户或网络暴露，需要加认证层。

## 3. 前端存储

**检查结果**：前端未使用 `localStorage` 或 `sessionStorage`。

- 聊天消息存在 JavaScript 数组 `messages[]` 中
- 章节上下文存为普通 JS 对象 `chapterContext`
- 刷新页面后所有状态清空
- 没有任何 token、key、敏感数据留在浏览器持久存储中

**结论**：无问题。

## 4. 依赖安全

### 4.1 已安装版本（当前环境）

| 包 | 版本 | 已知 CVE |
|----|------|----------|
| `fastapi` | 0.136.1 | 无已知严重 CVE |
| `uvicorn` | 0.30+（标准版） | 无已知严重 CVE |
| `httpx` | 0.28.1 | 无已知严重 CVE |
| `cryptography` | 48.0.0 | 无已知严重 CVE（42.x 系列的 CVE-2024-26130 已在 48.x 修复） |
| `pyyaml` | 6.0.3 | CVE-2020-14343 仅在 `yaml.load()` 未指定 Loader 时触发，本项目使用 `yaml.safe_load()` |
| `python-multipart` | 0.0.29 | 无已知严重 CVE |
| `aiofiles` | 25.1.0 | 无已知严重 CVE |

### 4.2 主要问题

| # | 问题 | 严重程度 | 说明 |
|---|------|----------|------|
| 1 | **依赖版本过于宽松** | 中低 | 所有依赖使用 `>=` 而非 `==`。新的 major/minor 可能引入行为变化或兼容问题。建议对生产依赖锁定版本 |
| 2 | **实际依赖多于声明** | 低 | `requirements.txt` 列了 7 个包，但运行时实际依赖 `pydantic>=2`、`sse-starlette>=2`（chat.py 用了 `sse_starlette.sse`）等。这些是 fastapi/uvicorn 的传递依赖，但显式声明是更好的实践 |
| 3 | **无自动安全扫描** | 低 | `pip-audit` 未安装，无法在 CI/部署前自动检测已知漏洞 |

## 5. 功能缺口

### 5.1 缺失功能汇总

| # | 缺失功能 | 影响 | 优先级 |
|---|----------|------|--------|
| 1 | **章节导出**（PDF/DOCX/MD） | 用户无法通过 API 将章节打包为交付格式。目前只有聊天记录可以 export Markdown | 中 |
| 2 | **全量备份** | 没有一键打包所有章节+知识库+状态的接口。如需备份只能逐个文件读取 | 低 |
| 3 | **批量操作** | 没有批量读取/写入/删除的端点。处理多文件场景需要发 N 次请求 | 低 |
| 4 | **知识库文件删除** | Knowledge 路由有 GET/POST，没有 DELETE。文件只能通过文件系统删除 | 低 |
| 5 | **日志记录** | api-design.md 规划了 logging（app.log、access.log、error.log），但未实现。目前无任何请求日志、错误追踪 | 中 |
| 6 | **请求限流** | 没有 rate limiting。本地工具威胁小，但如果未来暴露到局域网或公网，这一点值得考虑 | 低（当前） |
| 7 | **请求体大小校验** | `POST /knowledge/{filepath}` 没有显式限制写入内容大小。虽受 FastAPI 默认限制，但建议显式声明 | 低 |

### 5.2 架构性观察

- 没有用户认证模块——设计上假设本机运行。这是正确的权衡，但绑定 127.0.0.1 直接写在 `main.py` 中，没有通过环境变量配置
- CORS 全放通（`allow_origins=["*"]`）——对本机 SPA 无安全影响，但可能给扫描器留下印象"这里有宽松 CORS 配置"
- 所有数据持久化在 `data/` 目录，但 `data/` 不在 `.gitignore` 中——如果未来有人初始化 git 仓库时忘记添加 ignore，chat_history 等可能意外被 commit

## 7. 推荐修补项

**高优先级**（1-2 天内）：

1. `key_manager.py`：把静态 salt `b"shiyigongfang"` 改为随机生成 + 持久化在 `config.json` 的 `_salt` 字段中。这样每个安装实例的派生密钥唯一
2. `safe_write()`：添加文件大小校验，与 `safe_resolve()` 保持一致
3. `.gitignore`：追加 `data/` 目录，防止聊天记录和工作流状态意外进入版本控制

**中优先级**（后续迭代）：

4. `requirements.txt`：锁定主依赖版本（`cryptography==48.0.0` 等），避免意外兼容性破坏
5. `routers/knowledge.py`：添加 DELETE endpoint，实现知识库文件删除
6. 添加章节导出接口（POST /api/v1/chapters/export → 将选中的 md 合并打包为 docx）
7. 考虑移除 `routers/chat.py` 中 `_read_knowledge_file()` 的路径校验复制体，统一调用 `knowledge.py`

**低优先级**（随缘）：

8. 安装 `pip-audit` 并在启动脚本中加入依赖安全检测
9. `health` 路由中去除 `has_api_key` 暴露
10. 对 `list_files()` 中每次读取全文件做 CJK 计数的行为做缓存或延迟加载
